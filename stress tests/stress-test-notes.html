How it works

SheetJS reads the file in the browser and converts every tab into a JavaScript object — basically a grid of rows and columns in memory. That's the parse step.

The code then walks through that grid:

Step 1
Read the tab name
Identifies which program this class belongs to — Preschool, Level 1, Junior, etc.
→
Step 2
Scan the rows
Locates student names, skill names, and exactly where the score data starts in the grid.
→
Step 3
Read every cell
Each cell is either a 1 or blank. That value gets stored as the score for that student at that skill position.

That data gets sent to the Python backend, which takes the score data and program name, pulls the matching skill template, and builds the PDF using ReportLab — placing each star or circle at the exact x/y coordinate for that student column and skill row.

Two-part process

Frontend · JavaScript
SheetJS Parser
Reads the XLS, finds students and scores, identifies the program from the tab name.
Backend · Python
ReportLab Renderer
Takes the parsed data and draws the PDF — placing every symbol at the exact correct position.

Classes at the Same Time

When a time slot has too many students, the class gets split in iClassPro — Level 1 A, Level 1 B, Level 1 C — each a completely different group, all at the same time. The converter has to recognize each tab as its own class and render it on its own separate page.

Example
Preschool A | Monday | 3:30pm
Preschool B | Monday | 3:30pm
Preschool C | Monday | 3:30pm
Template
Preschool A | Day | #:##pm
Preschool B | Day | #:##pm
Preschool C | Day | #:##pm
Hiccup: When two classes run at the same time with no differentiator in the name, the system puts them on the same roll sheet. It uses the tab name as the unique ID for each page — so if two tabs have the exact same name, it treats them as one class.
Solution: Rename the classes in iClassPro with an A and B format — Preschool A, Preschool B — so each tab has a unique name. The code reads the full tab name as the page key, so A and B become two separate pages automatically.
How
File: public/index.html  ·  JavaScript  ·  SheetJS
wb.SheetNames.forEach(sheetName => { ... })

SheetJS reads every tab in the spreadsheet as a separate sheet. The code loops through every tab name and creates one page entry per tab — one tab = one PDF page. When you name them Preschool A and Preschool B in iClassPro, they export as two separate tabs. SheetJS hits Preschool A → creates page 1. Hits Preschool B → creates page 2. Done.

Class Being Over Capacity

When a class is booked over capacity, the PDF has to split across two pages. The cap is 6 students per page for Junior and Preschool, and 8 for Rec levels. If a class exceeds that, the converter automatically creates a second page and labels them Page 1 of 2 and Page 2 of 2.

Hiccup: There's no way to know how many students will push a class over capacity until the file is actually uploaded — so the overflow count isn't always predictable going in.
Solution: The system now knows the capacity per program and handles the split automatically — no manual intervention needed.
How
File: api/pdf_generator.py  ·  Python
capacity = prog.get('num_rows', 6)
chunks = [students[i:i+capacity] for i in range(0, len(students), capacity)]
if total_pages > 1:
    class_name = f"{class_name} (pg {chunk_idx+1}/{total_pages})"

The Python backend knows the max students per page for each program — 6 for Preschool and Junior, 8 for everything else. When it gets a class with more students than that, it slices them into chunks and generates one page per chunk. If there's more than one chunk, it automatically adds the (pg 1/2) and (pg 2/2) label to the class name. No manual setup required — it just handles it.

Accepted & Rejected Keywords

The converter reads the class name from the iClassPro tab name. It scans for keywords to figure out which program the class belongs to. If it finds a match — it processes it. If it doesn't — it skips it and shows a red warning.

Every keyword below is hardcoded in public/index.html. Adding or removing a keyword requires a code change. This is one of the things V2 will fix with the Keyword Manager.

Rejected — skipped with red warning

Class name containsResult
Ninja, Tumbling, Cheer, Xcel, Open Gym, Parent & Tot, Birthday, Camp — or anything not in the accepted list above⚠ Skipped — shown in red, not exported
Why this matters: If a gym renames a class — say "Recreational Level 1" instead of "Level 1" — it gets rejected. Every new naming format has to be added to the code manually. This is the #1 source of future maintenance work.
V2 fix: The Keyword Manager (Phase 2) moves all of this into a database table. Add a keyword, map it to a program, done — no code change needed.
